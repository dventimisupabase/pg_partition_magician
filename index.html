The trick

No data moves. At first.

The whole conversion is one well-behaved background job — not a maintenance window. Four moves:

I

Adopt

Rename the table aside, create the partitioned parent under the same name, and attach the original as the DEFAULT partition. Zero rows move; the app sees no change.

II

Premake

Keep partitions ready ahead of the write frontier, so every new row always has a real partition to land in.

III

Drain

Move the DEFAULT's closed tail into partitions in small batches. A CHECK-constraint trick keeps the work off the blocking lock — a 4-million-row attach drops from 101 ms to 0.4 ms.

IV

Maintain

Premake, drain, and retention run on a pg_cron schedule. Pause whenever you want — the DEFAULT partition keeps everything safe meanwhile.

Any monotonic key

Any key that grows with time.

If the partition key only moves forward — a clock, a sequence, a time-ordered id — pg_partition_magician can manage it.

♦

time

A timestamptz, timestamp, or date column. Monthly, daily, hourly — any interval.

select pgpm.adopt(
  'public.events',
  'created_at', '1 month');
♣

id

A bigint or numeric sequence — including Snowflake-style ids, where time lives in the high bits.

select pgpm.adopt_by_id(
  'public.events',
  'id', 10000000);
♠

uuidv7

Time-ordered uuids — and ULIDs stored as uuid. The grid is time; the bounds are encoded back into uuids.

select pgpm.adopt_by_uuidv7(
  'public.events',
  'id', '1 day');
No smoke, no mirrors

It tells you the truth.

The hard parts are documented, not hidden. The illusion is discipline, not deception.

One rule: monotonicity

The key must only move forward. Float keys are rejected outright — NaN and infinity would poison the frontier math.

Foreign keys, handled

An incoming FK can't survive a repartition silently, so adopt refuses by default — or drops and records them, and hands you a composite-FK recovery script.

A safety net, always

The DEFAULT partition stays. A row that arrives without a home is caught there, never lost — and surfaced by a check function.

Proven across versions

53 tests on PostgreSQL 15, 16, 17, and 18 — each installed three different ways, every partition kind exercised.
