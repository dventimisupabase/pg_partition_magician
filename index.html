Now a table. Now partitions.
It only looks like magic.

pg_partition_magician transmutes an existing Postgres table into a native RANGE-partitioned table, online. It attaches your table as the DEFAULT partition, keeps new partitions ahead of your writes, and drains the backlog in the background.

The reveal

Here's how the trick is done.

A good magician will, just once, turn the box around and show you the hinges. No data teleports; there's no secret copy. There are seven moves, and every one is plain Postgres, performed in the right order, on the right lock.

1 The setup

A live table can't become partitioned.

Postgres has no ALTER TABLE … PARTITION BY. The only built-in route is to create a fresh partitioned table and copy every row into it: a long lock, or a long migration with a risky cutover at the end. On a 200-million-row table that's the part nobody wants. pg_partition_magician never copies the table. Watch the hands.

2 The switch · transmute

The table you query is swapped for an identical one, mid-show.

transmute() renames your live table aside (events → events_default), creates a partitioned parent under the very same name, and attaches the old table to it as the DEFAULT partition. Same name, same rows. The app keeps querying events and never sees the hand move, and the swap is catalog-only, the lock held for a blink. What does change is small and declared: the partition key must already be part of the primary key (pgpm never rewrites it, move 3), non-key-aligned unique indexes can't be carried, and incoming foreign keys are kept against the new parent. If the table isn't shaped for that, transmute refuses up front and tells you how to fix it, rather than mangling it.

3 The index is already in hand

There's no hard part to hide.

A partitioned table's primary key must contain the partition key. pgpm's bet is that it already does, because the modern data model makes a time-ordered key (a Snowflake bigint, a UUIDv7, a ULID) the primary key. So there's nothing to widen and no index to rebuild: transmute reuses the existing key in place and the cutover is pure metadata, online at any size. If the partition key isn't part of your primary key, transmute says so up front and points you at the one-time migration, rather than quietly rebuilding the key under an ACCESS EXCLUSIVE lock (O(rows), about 28 minutes at 300M rows). Predictable beats clever.

4 The one rule of the act

Never touch the interval that's live.

Everything keys off one moving point: the frontier, the line between the settled past and the active present. Intervals behind it are closed: no new row will ever land there, so they're safe to rebuild into real partitions. The interval straddling it is open (writes are arriving now), and pgpm's single rule is to leave it alone. New rows fall into a premade future partition, or, if none fits yet, safely into DEFAULT. The frontier advances; the open interval closes; only then does it get reorganized.

5 The misdirection

Attaching a partition looks like it needs a full scan. It doesn't.

Attach a partition while DEFAULT holds data and Postgres scans the whole DEFAULT under ACCESS EXCLUSIVE (a hard lock) to prove no row belongs in the new partition. pgpm proves it the other way first: ADD CHECK (…) NOT VALID is catalog-only and instant, then VALIDATE runs the scan under SHARE UPDATE EXCLUSIVE, a gentle lock that lets reads and writes continue. With the proof in hand, the ATTACH skips its scan and is metadata-only. A 4-million-row attach drops from 101 ms to 0.4 ms; the cost moves to the lock nobody waits on.

6 Vanishing the backlog · drain

The old rows leave DEFAULT a chunk at a time.

The closed tail still sitting in DEFAULT is moved into its real partitions in small microbatches: DELETE … RETURNING a chunk, INSERT it where it belongs, commit, repeat. Background, paced, and pausable: DEFAULT shrinks toward empty while reads and writes never stop. Each newly-filled interval is attached by the misdirection above, so even that is a blink.

7 The act runs itself · maintenance_all

One pg_cron tick repeats every move, and shows its hands.

A single scheduled call ties it together: attain keeps the future stocked, drain empties the closed tail, retain drops cold partitions. It loops on pg_cron forever, and every action is written to an audit log in pgpm.log, the magician keeping his hands open. Pause it whenever; the DEFAULT keeps everything safe in the meantime.

Any monotonic key

Any key that grows with time.

If the partition key only moves forward (a clock, a sequence, a time-ordered id), pg_partition_magician can manage it.

♦

time

A timestamptz, timestamp, or date column. Monthly, daily, hourly: any interval.

select pgpm.transmute(
  'public.events',
  'created_at', interval '1 month');
♣

id

A bigint or numeric sequence, including Snowflake-style ids, where time lives in the high bits.

select pgpm.transmute(
  'public.events',
  'id', 10000000);
♠

uuidv7

Time-ordered uuids, and ULIDs stored as uuid. The grid is time; the bounds are encoded back into uuids.

select pgpm.transmute(
  'public.events',
  'id', interval '1 day');
No smoke, no mirrors

It tells you the truth.

The hard parts are documented, not hidden. The illusion is discipline, not deception.

One rule: monotonicity

The key must only move forward. Float keys are rejected outright: NaN and infinity would poison the frontier math.

Foreign keys, preserved

Because the partition key is part of your primary key and pgpm never rewrites it, the key an incoming FK references survives the conversion. Opt into p_incoming_fks => 'preserve' and transmute drops each incoming FK for the move, then re-adds it against the new parent once the drain is idle, no denormalization. A primary key that doesn't include the partition key is refused up front, never silently mangled.

A safety net, always

The DEFAULT partition stays. A row that arrives without a home is caught there, never lost, and surfaced by a check function.

Proven across versions

147 tests on PostgreSQL 15, 16, 17, and 18: each installed three different ways, every partition kind exercised.

Install

One file. Run it with psql.

Pure SQL, no compiled extension, no superuser. Point psql at the single source file and you're done:

psql "$DATABASE_URL" -f sql/pg_partition_magician.sql

No CLI? On Supabase or any SQL editor, the install page has a paste-and-run bundle. A Trusted Language Extension package is there too, for completeness.
